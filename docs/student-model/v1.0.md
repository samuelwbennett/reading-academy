# Student Model v1.0

**Authored by:** Agent 06 — Student Modeling
**Status:** v1.0, calibration-pending
**Date:** 2026-05-07

The per-student probabilistic layer that sits beneath the deterministic mastery state machine. The state machine answers "is this node mastered?" The model answers "how brittle is that mastery, when does this kid need to review, what's the choke point in their progression, and what should they do today?"

This is v1: intentionally simple, transparent formulas, no per-student parameter learning yet. Pilot data replaces the priors with measured curves in v1.1.

---

## Why a probabilistic layer

The mastery state machine is deterministic and binary: mastered or not. That's the right gate for progression. But for *scheduling* and *intervention*, binary is too coarse:

- A student who passed acquisition with median 2400ms and 90% accuracy is in the same state-machine bucket as one who passed with 1800ms and 95%. The first kid is fragile; the second is solid.
- A student who passed two months ago and hasn't seen the node since is in the same state-machine bucket as one who passed yesterday. The first needs urgent review; the second doesn't.
- A student stuck on `DG_sh` looks like ten possible bottleneck stories: weak phonics? weak PA prereq? not enough sleep? gaming the system? The state machine can't tell.

The model doesn't replace the state machine. It augments it. The state machine still decides when a node advances; the model decides what to do *within* a node, *across* nodes, and *across days*.

## Mastery probability — `pCorrect`

The probability that the next attempt on this node will be correct. This is **not** the state-machine output; it's the underlying continuous estimate.

v1 formula (simple Beta-Bernoulli with latency penalty):

```python
def p_correct(node_state, node_config):
    attempts = node_state.attempts[-30:]  # last 30 attempts
    if not attempts:
        return 0.0
    
    # Beta posterior with weak prior
    alpha_prior, beta_prior = 1, 1   # uniform prior
    alpha = alpha_prior + sum(1 for a in attempts if a.correct)
    beta  = beta_prior  + sum(1 for a in attempts if not a.correct)
    p_acc = alpha / (alpha + beta)
    
    # Latency penalty: median latency above acquisition gate reduces confidence
    latencies = [a.latencyMs for a in attempts if a.correct]
    if latencies:
        median_lat = sorted(latencies)[len(latencies)//2]
        target = node_config.mastery.read_latency_ms
        latency_factor = min(1.0, target / max(median_lat, 1))
    else:
        latency_factor = 0.5
    
    # Decay by recency — recent attempts weight more
    days = days_since_last_attempt(node_state)
    recency_factor = math.exp(-days / 14)  # half-life ~10 days
    
    return p_acc * (0.5 + 0.5 * latency_factor) * (0.5 + 0.5 * recency_factor)
```

Key properties:

- New student, zero attempts: returns 0 (correct — no evidence of skill).
- 10/10 correct, no latency penalty, today: returns ~0.92.
- 10/10 correct, 30 days ago: returns ~0.30 (forgetting decay).
- 8/10 correct, fast: returns ~0.65.
- 8/10 correct, slow: returns ~0.40.

The 14-day half-life is a prior — pilot data replaces it with a per-node calibrated value.

## Retrievability and stability — FSRS-lite

FSRS (Free Spaced Repetition Scheduler) is the open-source successor to SuperMemo's spaced-repetition algorithms. The full FSRS has 17 trainable parameters; v1 uses a 3-parameter approximation that captures the load-bearing dynamic without the calibration overhead.

The core idea: each node has a **stability** (days until retrievability drops to 90%) and a **retrievability** (probability of recall right now).

```python
import math

# Initial stability for a freshly mastered node, by item type.
INITIAL_STABILITY_DAYS = {
    "phoneme_*":   3.0,
    "letter_sound": 2.5,
    "read_aloud":  4.0,
    "cold_passage": 7.0,
}

def retrievability(stability_days, days_since_review):
    """FSRS retrievability formula: R(t) = (1 + t / (9 * S)) ** -1"""
    if stability_days <= 0:
        return 0.0
    return (1 + days_since_review / (9 * stability_days)) ** -1

def update_stability(prev_stability, recall_success, recall_speed_factor):
    """Update stability after a review attempt.
       Successful, fast review → big stability gain.
       Successful, slow review → small stability gain.
       Failed review → stability halves."""
    if not recall_success:
        return max(0.5, prev_stability * 0.5)
    
    # Successful review: stability grows multiplicatively.
    # Speed factor is acquisition_target_latency / actual_latency, capped 0.3..2.0.
    growth = 1.5 + 1.5 * (recall_speed_factor - 0.5)
    return prev_stability * growth
```

Per-node behavior in practice:

| Days since review | Stability = 3 days | Stability = 7 days | Stability = 30 days |
|---|---|---|---|
| 1 | R = 0.96 | R = 0.98 | R = 0.996 |
| 7 | R = 0.79 | R = 0.90 | R = 0.974 |
| 30 | R = 0.47 | R = 0.68 | R = 0.90 |
| 90 | R = 0.23 | R = 0.41 | R = 0.75 |

The `nextReviewDue` for a node = the future date at which retrievability is predicted to drop to 0.85. This is what feeds the spaced-review schedule that Reading Fluency owns.

```python
def days_to_review(stability_days, target_retrievability=0.85):
    """Solve R(t) = target for t."""
    return 9 * stability_days * (1 / target_retrievability - 1)
```

For stability=3, target=0.85: ~4.8 days. For stability=7: ~11 days. For stability=30: ~48 days.

## Fragility score

The composite metric that says "this mastery is brittle." Lower is better.

```python
def fragility(node_state, node_config):
    attempts = node_state.attempts[-30:]
    if len(attempts) < 5:
        return 0.5  # insufficient data; assume moderate
    
    correct = [a for a in attempts if a.correct]
    if not correct:
        return 1.0
    
    # 1. Latency variance: std-dev / mean of correct-attempt latencies.
    lats = [a.latencyMs for a in correct]
    mean = sum(lats) / len(lats)
    var = sum((l - mean) ** 2 for l in lats) / len(lats)
    std = var ** 0.5
    cv = std / max(mean, 1)         # coefficient of variation
    f_latency = min(1.0, cv * 2)    # cv > 0.5 → fully fragile
    
    # 2. Encoding-decoding gap (when both halves exist).
    read_correct  = [a for a in attempts if a.itemKind == "read_aloud" and a.correct]
    spell_correct = [a for a in attempts if a.itemKind == "dictation"  and a.correct]
    if read_correct and spell_correct:
        read_acc  = len(read_correct)  / sum(1 for a in attempts if a.itemKind == "read_aloud")
        spell_acc = len(spell_correct) / sum(1 for a in attempts if a.itemKind == "dictation")
        f_gap = min(1.0, max(0, read_acc - spell_acc) * 2)
    else:
        f_gap = 0.0
    
    # 3. Recent error streak relative to current accuracy.
    recent_5 = attempts[-5:]
    recent_errors = sum(1 for a in recent_5 if not a.correct)
    f_streak = recent_errors / 5
    
    # 4. Days since last attempt (forgetting risk).
    days = days_since_last_attempt(node_state)
    f_recency = min(1.0, days / 30)
    
    # Weighted sum.
    return (
        0.30 * f_latency +
        0.20 * f_gap +
        0.30 * f_streak +
        0.20 * f_recency
    )
```

A node is **flagged fragile** when `fragility >= 0.5`. Flagged fragile nodes get prioritized for review in the daily session manifest, even if the state machine still says they're mastered.

## Bottleneck detection

A bottleneck is a node that's blocking forward progress. Algorithm:

```python
def find_bottlenecks(state, node_defs):
    bottlenecks = []
    for node in node_defs:
        ns = state.nodes[node.id]
        # Bottleneck conditions:
        # - Status is "practicing" (window full but not mastered)
        # - At least 20 attempts logged
        # - p_correct < 0.6
        if (
            ns.status == "practicing" and
            len(ns.attempts) >= 20 and
            ns.model.pCorrect < 0.6
        ):
            bottlenecks.append({
                "nodeId": node.id,
                "pCorrect": ns.model.pCorrect,
                "attemptsToDate": len(ns.attempts),
                "blockedDescendants": count_locked_descendants(node, node_defs, state)
            })
    
    # Sort by impact: blocked descendants first, then by attempts (more time spent = bigger problem).
    bottlenecks.sort(key=lambda b: (-b["blockedDescendants"], -b["attemptsToDate"]))
    return bottlenecks
```

Output is consumed by Intervention agent (#7) to surface to teachers/parents. The model itself doesn't decide what to *do* about a bottleneck — that's an Intervention-layer decision.

## Velocity model

Captures progression rate over time, used by the orchestrator to gauge fatigue and growth:

```python
def velocity(state, node_defs, window_days=14):
    cutoff = now() - window_days * 86400_000
    
    # Items per minute (recent rolling)
    recent_attempts = [a for nid, ns in state.nodes.items() for a in ns.attempts if a.ts >= cutoff]
    if not recent_attempts:
        return { "itemsPerMinute": 0, "nodesPerWeek": 0, "trend": 0, "consistency": 0 }
    
    sessions = group_by_session(recent_attempts)
    total_items = sum(len(s) for s in sessions)
    total_minutes = sum(session_duration_min(s) for s in sessions)
    items_per_min = total_items / max(total_minutes, 1)
    
    # Nodes mastered per week
    masteries = [ns.masteredAt for ns in state.nodes.values()
                 if ns.masteredAt and ns.masteredAt >= cutoff]
    nodes_per_week = len(masteries) / (window_days / 7)
    
    # Trend: regression slope of weekly velocity over 4 weeks
    trend = compute_trend(state, weeks=4)
    
    # Consistency: fraction of last 14 days with at least one session
    days_active = count_unique_active_days(recent_attempts, window_days)
    consistency = days_active / window_days
    
    return { "itemsPerMinute": items_per_min, "nodesPerWeek": nodes_per_week,
             "trend": trend, "consistency": consistency }
```

The trend signal is what tells the orchestrator "this kid is plateauing — back off the new lessons" or "this kid is accelerating — give more challenge."

## Fatigue detection (within a session)

Fatigue is short-horizon: it's about the current session, not the multi-week trend.

```python
def fatigue_score(session_state):
    """0 = fresh, 1 = exhausted."""
    score = 0.0
    
    # Duration: linear ramp after 15 min.
    minutes = session_state.minutes_elapsed
    score += min(1.0, max(0, (minutes - 15) / 10)) * 0.3
    
    # Recent error streak.
    recent_attempts = session_state.attempts[-8:]
    if len(recent_attempts) >= 4:
        recent_errors = sum(1 for a in recent_attempts if not a.correct)
        score += (recent_errors / 8) * 0.4
    
    # Latency creep: latencies later in session > earlier.
    if len(session_state.attempts) >= 12:
        early = mean(a.latencyMs for a in session_state.attempts[:6])
        late  = mean(a.latencyMs for a in session_state.attempts[-6:])
        if late > early * 1.4:
            score += 0.3
    
    return min(1.0, score)
```

Fatigue ≥ 0.6: orchestrator shortens the session or substitutes review for new lesson.
Fatigue ≥ 0.85: orchestrator suggests stop ("Great work today — see you tomorrow").

## Anomaly detection (conservative)

The model flags two kinds of anomalies:

1. **Lucky drill / outlier session.** A session whose WCPM is > 1.5× the student's rolling mean *and* > 1.2× the prior personal best. Doesn't invalidate the score; just flags for the parent dashboard so a 35→55 WCPM jump doesn't generate "your child improved 50% overnight" claims.

2. **Possible gaming.** A pattern of very fast incorrect answers followed by a slower correct answer on the same item — suggests guess-and-retry rather than skill. Detected via per-item attempt sequences.

Operating principle 8: conservative on anomaly flags. Both detectors require multiple sessions of evidence before firing.

## What the model does NOT do (yet)

Explicit non-goals for v1:

- **Learn per-student parameters.** All formulas use static priors. Per-student calibration arrives in v1.1 when pilot data is available.
- **Use neural-net or LLM-based prediction.** v1 is logistic + linear. Sophistication is not the bottleneck; data is.
- **Cross-app modeling.** Reading Academy + Reading Facts + Math Academy as a unified model. Phase 12 (Cross-App Orchestration agent) territory.
- **Cohort-relative norms.** Until N>60, "this student is in the 30th percentile" is statistical fiction. Hold.
- **Predict optimal interventions.** Bottleneck detection surfaces problems; what intervention to prescribe is Intervention agent (#7) territory.

## Calibration plan

When pilot data is available (target N=60 students, 6 weeks of usage):

1. Replace `INITIAL_STABILITY_DAYS` with empirically-fit values per node type.
2. Replace the 14-day half-life recency factor with per-node decay constants.
3. Replace fragility weights (0.30/0.20/0.30/0.20) with regression-fit weights against measured "did this mastered node regress within 30 days" outcome.
4. Recalibrate `p_correct` Beta-Bernoulli prior strength based on observed predict/actual accuracy.

Calibration is a v1.1 deliverable.
