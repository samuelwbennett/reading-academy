```
└─────────────────────────────┬────────────────────────────────────┘

                              ▼
┌──────────────────────────────────────────────────────────────────┐
│                    SUPABASE (Postgres)                           │
│                                                                  │
│   learning_apps · students · student_app_accounts                │
│   practice_sessions · skill_attempts · daily_progress · goals    │
│                                                                  │
│   RLS-gated; anon key from browser, service-role from functions  │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│              VPA ORCHESTRATION LAYER (separate SPA)              │
│   Consumes /api/snapshot + /api/mastery from each app            │
│   Renders cross-app dashboard                                    │
└──────────────────────────────────────────────────────────────────┘
```

**Three layers:** browser (SPA + pure engines), edge (stateless aggregators), database (durable state). Pedagogy logic lives in the browser as pure functions; persistence and aggregation live at the edge and DB.

**Why local-first:** the orchestrator + state machine must run with sub-100ms decisions. A network round-trip per attempt destroys the UX. Browser is the primary execution environment; Supabase is durable replication, not real-time control.

## 2. Runtime data flow

### Single attempt (the hot path)

```
1. UI renders item from current Task slot.
2. Student responds (mic, tap, type).
3. ASR/input layer produces an Attempt object.
4. masteryEngine.recordAttempt(state, nodeId, attempt) → newState.
5. Telemetry event emitted to localStorage queue.
6. UI renders next item from current Task slot.

Asynchronously (debounced 2s):
7. Storage layer writes newState to localStorage (write-through cache).
8. Storage layer flushes telemetry queue to Supabase via REST.
9. Student model recomputes lazily (next session-start).
```

**The hot path never blocks on the network.** All four critical operations (record, score, advance, render) happen against in-memory state. Network is best-effort replication.

### Session start (cold path)

```
1. SPA loads, reads state from localStorage.
2. If online and signed-in: pull latest from Supabase, merge (last-write-wins on per-attempt timestamps).
3. computeStudentModel(state) → model.
4. composeSession(state, model, nodeDefs) → SessionManifest.
5. UI renders Today's Tasks.
```

### Session end

```
1. UI commits final state to localStorage.
2. UI flushes telemetry queue.
3. UI POSTs aggregate session record to Supabase practice_sessions.
4. /api/snapshot is now reading fresh data on next dashboard query.
```

### Cross-app dashboard query

```
1. VPA orchestration layer calls https://reading-academy.vercel.app/api/snapshot?student=<uuid>.
2. Vercel function reads learning_apps + student_app_accounts.state from Supabase.
3. Computes today's XP, week XP, next drill, returns JSON.
4. Orchestration layer renders.
```

## 4. Engineering sequence

Five milestones to pilot. Each is a working artifact, not a spec.

### M1 — Vertical slice (target: 1–2 weeks engineering)

**Goal:** A student can sign in, take placement, complete one full daily session (lesson + Reading Facts + maybe passage), and have all data flow to Supabase. No item-bank diversity yet; minimum content to prove the loop.

**Deliverables:**
- Port full 55-node graph from `docs/curriculum/` to `src/data/skill_nodes.json` (replace 5-node subset)
- Port FL_01 passage bank from `docs/passages/bank/FL_01/` to `src/data/passages.json` via concat script
- Implement `min_items` enforcement in `evaluateMastery`
- Implement state machine T5 (handoff to Reading Fluency) in `recordAttempt`
- Implement `src/lib/readingFactsEngine.js` with `pickFocal`, `scoreItem`, `scoreDrill`, `evaluateRetirement`, `applyTrickleDown`
- Implement `<ReadingFactsDrill>` component
- Implement `src/lib/studentModel.js` per `docs/student-model/v1.0.md`
- Implement `src/lib/sessionOrchestrator.js` per `docs/student-model/daily-session-orchestrator-v1.0.md`
- Replace stub `getTodayTasks` callsite with `composeSession` output
- Implement `src/lib/passageRecommender.js`
- Implement `<PassageReader>` component (cold-passage drill UI)
- Implement Levenshtein word-aligner for cold-passage scoring
- Wire telemetry events emission throughout (see §7)
- Wire telemetry write-through to Supabase

**Exit criteria:**
- A student can complete a 17-min session containing all 4 task types (lesson, Reading Facts, review, passage)
- All telemetry lands in `skill_attempts` correctly
- `/api/snapshot` returns sane JSON for that student
- `/api/mastery` returns sane JSON
- Build passes; no console errors

### M2 — Item & passage authoring (target: 2–3 weeks)

**Goal:** Item banks at 20+ items per node; passage banks at 6 per fluency gate.

**Deliverables:**
- Author 1100+ items across 55 nodes (~20 items/node)
- Author 18 more passages (FL_02, FL_03, FL_04 banks at 6 each)
- Validator script for items + passages, run in CI
- Item-authoring guide (Agent #3 ships this)

**Exit criteria:**
- Validator passes 100% of authored content
- A student can encounter genuinely cold items in mastery checks
- All four fluency gates have functional passage banks

### M3 — Auth + orchestration integration (target: 1–2 weeks)

**Goal:** Real Supabase auth, real orchestration-layer adapter.

**Deliverables:**
- Magic-link sign-in flow on first run
- Anonymous session → claimed-by-auth migration logic
- `learning_apps` row inserted (slug = `reading_academy`)
- Orchestration adapter shipped in `vpa-orchestration-layer` repo
- `/api/snapshot` and `/api/mastery` deployed and responding for real students

**Exit criteria:**
- A student signs in once; subsequent visits resume state
- Orchestration dashboard shows Reading Academy alongside Reading Facts and Math Academy

### M4 — Pilot hardening (target: 1–2 weeks)

**Goal:** Production-grade reliability for 60+ students.

**Deliverables:**
- Error boundaries and recovery UI
- Offline-first robustness (mid-session crash recovery, sync-on-reconnect)
- Parent dashboard (lite — single screen showing trend)
- Teacher roster view (basic)
- Telemetry monitoring dashboard
- Load test with synthetic 60-student concurrent traffic
- Incident-response runbook

**Exit criteria:**
- 60-student concurrent test passes without errors
- Mid-session refresh restores correct state
- Teacher can see roster and per-student WCPM trend

### M5 — Pilot launch (target: 1 week launch + 6 weeks running)

**Goal:** 60 K-2 students, 4 days/week, 17 min/day, 6 weeks.

**Deliverables:**
- Onboarding videos (parents, teachers)
- Weekly progress emails
- Pre/post DIBELS NWF + ORF assessment coordination
- Weekly cohort review with Student Modeling agent
- Issue triage rotation

**Exit criteria:**
- ≥70% of students complete ≥80% of assigned sessions
- Median NWF gain ≥15 correct letter-sounds/min
- Median ORF gain ≥12 WCPM
- ≥60 students remain enrolled at week 6

## 7. Telemetry architecture

### Event catalog (frozen v1.0)

Every event has the shape: `{ event: string, ts: number, studentId?: string, ...fields }`.

**Pedagogy events (consumed by Student Modeling agent):**

```
mastery.attempt          { nodeId, itemId, prompt, recognized, correct, latencyMs, drillId? }
mastery.transition       { nodeId, from, to, reason }
mastery.regression       { nodeId, from, automaticAt, regressedAt }
fluency.attempt          { nodeId, word, latencyMs, recognized, correct, withinTarget, asr }
fluency.drill_complete   { drillId, durationMs, wordsAttempted, wordsCorrect, wordsAutomatic, wcpm, personalBest, nodesTouched, trickleDownCredits }
student_model.updated    { diff }
session_manifest.committed { manifest }
session.started          { sessionId, manifestId }
session.ended            { sessionId, durationMs, completion: 0..1, fatigueScore }
```

**Process events (consumed by ops):**

```
build_milestone.completed   { milestone, deliverables, qaSignoff }
schema.frozen               { schemaName, version }
auth.signin                 { method }
asr.fallback_to_adult       { reason }
```

### Pipeline

```
[component event]
       ↓
   telemetry.emit(event)
       ↓
   in-memory queue (capped 200)
       ↓
   ─── flushed every 5s OR on session.ended ───
       ↓
   batched POST to Supabase REST
       ↓
   skill_attempts (one row per attempt-style event)
   practice_sessions (one row per session.ended)
   daily_progress (upserted from event aggregate)
```

### Persistence schema in Supabase

`skill_attempts` carries the granular events (existing universal table). `practice_sessions` carries session-level aggregates. `daily_progress` is the rolled-up view that `/api/snapshot` reads.

The telemetry library (`src/lib/telemetry.js`) is the only thing that writes to Supabase from the SPA. Components don't make REST calls directly.

## 8. Session runtime lifecycle

```
[user opens app]
       │
       ▼
loadState() → state from localStorage
       │
       ├── if state.diagnosticComplete = false:
       │      → render <Diagnostic />
       │      → on complete: applyDiagnostic(state, results) → state'
       │
       ▼
computeStudentModel(state, nodeDefs) → model
       │
       ▼
composeSession(state, model, nodeDefs) → manifest
       │
       ▼
emit('session_manifest.committed', { manifest })
       │
       ▼
render <TodayTasks tasks={manifest.tasks} />
       │
       ▼
[user taps a task card]
       │
       ▼
emit('session.started', { sessionId })
       │
       ▼
for each task in manifest:
   render the appropriate component:
     - lesson  → <Drill />
     - review  → <Drill />
     - fluency → <ReadingFactsDrill />
     - passage → <PassageReader />
       │
       ▼
   each item:
     - render
     - capture response (via MicButton or input)
     - score (pure fn)
     - emit('mastery.attempt' or 'fluency.attempt')
     - recordAttempt(state, ...) → state'
     - if state machine transitioned: emit('mastery.transition')
     - render next item
       │
       ▼
   each task end:
     - emit('fluency.drill_complete' or other aggregate)
       │
       ▼
[all tasks complete OR user exits]
       │
       ▼
emit('session.ended', { ... })
       │
       ▼
saveState(state) to localStorage
       │
       ▼
flushTelemetry() → batched POST to Supabase
       │
       ▼
return to dashboard, manifest re-evaluates if midnight passed
```

The lifecycle is **reentrant**: a refresh mid-session reads `state` from localStorage, recomputes manifest (or restores in-progress manifest from local cache), and resumes where left off.

## 9. Supabase schema plan

### Use the universal multi-tenant tables

Reuse the existing schema from `001_initial_schema.sql`. Reading Academy adds **one row** to `learning_apps`, no new tables. All Reading Academy state lives in:

- `students` — auth-linked profiles
- `student_app_accounts.state` (JSONB) — the per-student blob containing `nodes`, `model`, `fluency`, `diagnosticComplete`
- `practice_sessions` — one row per completed session, aggregate
- `skill_attempts` — granular per-attempt events
- `daily_progress` — rolled-up daily metrics (upserted by Vercel functions)
- `goals` — daily XP target overrides

### `learning_apps` row (insert at M3)

```sql
insert into learning_apps (slug, name, snapshot_url, mastery_url, color)
values (
  'reading_academy',
  'Reading Academy',
  'https://reading-academy.vercel.app/api/snapshot',
  'https://reading-academy.vercel.app/api/mastery',
  '#0a84ff'
);
```

### `student_app_accounts.state` JSONB shape

```json
{
  "diagnosticComplete": true,
  "nodes": {
    "PA_04_blend_cvc": {
      "status": "automatic",
      "attempts": [...],
      "masteredAt": 1728492000000,
      "automaticity": {...},
      "model": {...}
    }
  },
  "fluency": {
    "personalBests": {...},
    "rollingDailyWcpm": [...]
  },
  "model": {
    "velocity": {...},
    "fragility": {...},
    "lastUpdated": 1728492000000
  },
  "currentManifest": {...}
}
```

This is one document per student per app. Reads are by `(student_id, app_id)`; writes are atomic JSON updates. The browser is the source of truth during a session; the row is the durable replica.

### RLS policies

- Anon key: read-only access to `learning_apps`. Cannot read `students` or `student_app_accounts` directly.
- Authenticated user: read+write access to their own `students` row and their own `student_app_accounts` row only.
- Service role (functions only): read-only access to all `student_app_accounts.state` for snapshot/mastery aggregation.

### No new tables for M1

No `reading_*` app-specific tables. The universal schema is sufficient for the K-2 phase. App-specific tables become a temptation when extending; resist until clearly justified by query patterns.

## 11. Backend responsibilities

### Vercel serverless functions

- `/api/snapshot` — read student state, compute today's XP + next drill, return JSON. Stateless. Read-only.
- `/api/mastery` — read student state, compute strand-level rollup, return JSON. Stateless. Read-only.
- `/api/azure-pa-proxy` (M1.5+) — accept audio + expected text, forward to Azure with server-side key, return per-phoneme scores.

**Functions never write to Supabase.** All durable writes happen from the SPA via the Supabase JS client. Functions are read-aggregators with one exception: the Azure proxy may write request logs for observability.

### Supabase

- Source of truth for durable state.
- RLS enforces per-user isolation.
- No business logic in Postgres functions or triggers (resist the temptation; everything stays in the SPA + lib layer).

### Vercel hosting

- Build: `npm run build` → `dist/`
- Static assets served from CDN
- Functions auto-deployed from `api/`
- Env vars: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `AZURE_PA_KEY` (M1.5+)

### What does NOT live on a server

- Pedagogy logic (mastery state machine, scoring, orchestration, Reading Facts engine, student model). All in browser.
- Item/passage authoring data. Static JSON, baked into the build.
- ASR (other than Azure proxy). Web Speech API runs in browser.

## 12. State machine integration

The mastery state machine (Agent #3) is implemented in `src/lib/masteryEngine.js`. Eight states, ten transitions. The integration points:

```
┌──────────────────────────┐
│ recordAttempt(state, ...)│ ← called from each task component
└──────────────┬───────────┘
               ▼
       evaluateMastery() → are we at T4?
               ▼
       cascadeUnlock() → are any other nodes at T1?
               ▼
       if T4 fired: trigger T5 (handoff to Reading Fluency)
               ▼
       readingFactsEngine.onNodeMasteredForAcquisition(nodeId)
               │
               └── enters node into automaticity zone
               ▼
       emit telemetry events for any transition that fired
               ▼
       return newState
```

**M1 engineering tasks (state machine):**

## 13. Reading Facts integration

Reading Facts is a **separate task type** from lesson/review. It has its own component, its own scheduler, and its own scoring. Integration points:

### Scheduler hook

When `composeSession()` runs, it calls `readingFactsEngine.pickFocal(state, nodeDefs)` to get today's focal RF node. The orchestrator places 3 RF drill slots in the manifest (per `daily-session-orchestrator-v1.0.md`).

### Drill execution

`<ReadingFactsDrill>` is invoked from the Drill view when the active task type is `fluency`. It:

1. Receives `focalNodeId` from the manifest.
2. Builds an item sequence using the 70/30 interleave rule.
3. Runs the 60-second timed loop.
4. Emits `fluency.attempt` per word and `fluency.drill_complete` at the buzzer.
5. Calls `readingFactsEngine.evaluateRetirement(focalNodeId, drillScore)` to handle T6/T7.

### State updates

The Reading Facts engine writes to `state.nodes[nodeId].automaticity` directly. It never touches `acquisition` state — that's still owned by the mastery engine.

1. Implement `pickFocal` per `reading-facts-engine-v1.0.md` §scheduler. ~40 lines.
2. Implement `scoreItem`, `scoreDrill`. ~30 lines.
3. Implement `evaluateRetirement`, `applyTrickleDown`. ~50 lines.
4. Build `<ReadingFactsDrill>` component. ~150 lines.
5. CSS for the drill (timer, mic, end screen). ~80 lines.

Total: ~350 lines. None of this duplicates existing Drill component logic — the timing and item-flow are different enough to warrant separation.

## 14. Passage system integration

### Build-time concatenation

The passage banks at `docs/passages/bank/<gate>/passages.json` are not directly imported. A build script at `scripts/build-passages.js` walks the directory and concatenates into `src/data/passages.json`.

```js
// scripts/build-passages.js (sketch)
import { readdir, readFile, writeFile } from "fs/promises";
import { join } from "path";

const banks = await readdir("docs/passages/bank");
const all = [];
for (const bank of banks) {
  const file = await readFile(join("docs/passages/bank", bank, "passages.json"), "utf8");
  const json = JSON.parse(file);
  all.push(...json.passages.filter(p => p.validationStatus === "passed"));
}
await writeFile("src/data/passages.json", JSON.stringify({ passages: all }, null, 2));
```

Run as a `prebuild` npm script. CI runs it automatically.

### Recommender integration

`src/lib/passageRecommender.js` exports `recommendPassages(request, allPassages, nodeDefs)` per the contract in `passage-architecture-v1.0.md` §Recommender.

Called from:
- Session orchestrator when composing a passage slot (via `intent: "lesson_practice"`).
- `<PassageReader>` when fetching the cold passage for a fluency-gate attempt (`intent: "cold_gate_attempt"`).

### `<PassageReader>` component

UI is a passage rendered with sentence breaks, a mic button, and a 60-second timer (for fluency gates) or untimed (for practice passages). Scoring:

1. Continuous ASR captures audio for the full reading.
2. At end (timer or "I'm done" button), final transcript is aligned to expected text via Levenshtein word-aligner.
3. WCPM = correctly-aligned words / minute.
4. Accuracy = correctly-aligned / total attempted.
5. Pass/fail computed against the gate's `mastery.wcpm_min` and `passage_accuracy`.
6. Emit `mastery.attempt` per gate node with the cold-passage result.

### M1 engineering tasks (passages):**

1. Build script `scripts/build-passages.js`. ~30 lines.
2. `passageRecommender.js`. ~80 lines.
3. Levenshtein word-aligner. ~50 lines.
4. `<PassageReader>` component. ~150 lines.

Total: ~310 lines.

## 15. Orchestrator runtime integration

The session orchestrator (`src/lib/sessionOrchestrator.js`) is the **single point of truth** for "what does this kid do today." Everything else feeds it; nothing routes around it.

### Inputs

```ts
composeSession(state: StudentState, model: StudentModel, nodeDefs: NodeDef[]): SessionManifest
```

Pure function. Same inputs → same output. Fully testable.

### Outputs

A `SessionManifest` object (shape in `daily-session-orchestrator-v1.0.md`). Persisted to `state.currentManifest` and rendered by `<TodayTasks>`.

### Recompute triggers

The manifest is computed:

1. On session start (cold path, app loads).
2. After midnight Denver time (manifest stale).
3. After any mastery transition (the new state may unlock different work).
4. When fatigue ≥ 0.6 mid-session (orchestrator may swap remaining slots to lighter work).

### Mid-task immutability

The current manifest does not change while a task is in progress. A student doing lesson item 3 doesn't have the lesson swap underneath them. Only between tasks does the orchestrator re-evaluate.

### Integration with current MVP

Replace `getTodayTasks` callsite in `App.jsx`:

```js
// Before:
const tasks = getTodayTasks(state, nodes, 4);

// After:
const manifest = useMemo(() => composeSession(state, model, nodes), [state, model, nodes]);
const tasks = manifest.tasks;
```

Test framework: Vitest (Vite-native; minimal config).

### Integration tests (M1)

End-to-end scenarios scripted with Playwright:

1. New student → diagnostic → first session → ≥1 mastery transition.
2. Returning student → state restored → manifest correct.
3. Mid-session refresh → state preserved → resumed at right slot.
4. Microphone denied → adult-fallback UI appears → drill still completable.
5. Offline → app loads from localStorage → telemetry queues → online → flushes.

### Validator (M2)

CI script `scripts/validate-content.js` runs on every PR:

- All passages pass `validatePassage`.
- All items have valid item types and `expected` fields.
- All node IDs referenced in items/passages exist in skill_nodes.json.
- No orphaned references; no cycles in skill graph.

PR fails if validator fails.

### Manual QA (M4)

Before pilot launch:

- Full cold-passage attempt with adult observer for 5 students at varying levels.
- Consistency check: two evaluators score the same student session manually; must agree on mastery state for every node.
- Latency check: response time from item-render to next-item-render measured in Chrome DevTools; target < 200ms.

### Telemetry validation (continuous)

Daily cron query:

- Are events landing in `skill_attempts` at the expected rate (~50 events per active student per day)?
- Is the `event` field always one of the catalog values (no typos, no untyped events)?
- Is `ts` monotonic and within Denver-day bounds?

Anomalies trigger alerts.

## 19. Scope-control rules

For M1 only. **In** means it ships; **out** means it does not, and adding it requires this agent's approval + reasoning about why pilot can't wait.

### IN

- Mastery state machine (8 states)
- Reading Facts engine (60s drills, 70/30 interleave, retirement, trickle-down)

- Web Speech API ASR
- localStorage + Supabase write-through
- `/api/snapshot` and `/api/mastery`
- XP system with daily 30 target
- Course tree UI with hover-expand
- Today's tasks card with task pills
- Parent/teacher dashboard (lite — single screen, WCPM trend)

### OUT

- Azure ASR (M1.5)
- Adaptive diagnostic (sequential is enough)
- All G3+ content (morphology, comprehension, knowledge arcs)
- Spanish/ELL track
- Avatars, badges, streaks (Math Academy doesn't have them; we don't either)
- AI tutor chat
- Video lessons
- Native mobile app

- Offline mode beyond single session
- Per-phoneme scoring
- Prosody scoring
- Comprehension questions
- Personalized item generation

### Anti-goals

- **Engagement over learning.** Don't add a feature because it might make the app "more fun" if there's no evidence it improves outcomes.
- **Sophistication theater.** Don't add a sophisticated model when a simple one is calibratable on the data we have.
- **Cargo-culting Math Academy.** Math Academy got there through 10 years of curriculum work. We mirror their *patterns* (mastery, knowledge graph, XP, no badges) but not their *scale* — we ship 55 nodes, not 1500.
